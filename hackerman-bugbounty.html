// THE REAL BUG BOUNTY WORKFLOW

This is how actual hunters operate. Follow this order every time — skip steps and you'll miss bugs or get out-of-scope.

1
READ THE SCOPE. SERIOUSLY.

Before touching anything — read the entire program brief on HackerOne / Bugcrowd. Know exactly what's in scope (domains, endpoints, apps) and what's out (admin panels, third-party services, rate limiting).

Out-of-scope reports = instant reject + possible ban. Programs list their scope at hackerone.com/[company]

RULE #1Get written/implied auth via program
2
PASSIVE RECON — DON'T TOUCH YET

Gather intel without hitting their servers. Use OSINT tools to map the attack surface.

  • ▸ whois domain.com — registrar, nameservers
  • ▸ shodan.io — find exposed services, ports, certs
  • ▸ crt.sh/?q=%.domain.com — SSL cert subdomain leak
  • ▸ web.archive.org — old endpoints, JS files, secrets
  • ▸ github.com/search?q=domain.com — leaked keys, tokens
3
ACTIVE RECON — ENUMERATE

Now you start touching servers. Enumerate subdomains, ports, directories. Be stealthy — don't hammer with huge wordlists unless the program allows aggressive testing.

  • ▸ Subdomain enum: subfinder, amass, assetfinder
  • ▸ Alive check: httpx — filter live hosts
  • ▸ Port scan: nmap -sV -T3 -p- target.com
  • ▸ Dir fuzz: ffuf -u https://target.com/FUZZ -w wordlist.txt
  • ▸ JS recon: gau, waybackurls for old endpoints
4
FINGERPRINT & MAP

Figure out what's running. Tech stack, frameworks, API structure, auth mechanisms. This tells you WHERE to look for bugs.

  • ▸ whatweb / Wappalyzer browser extension
  • ▸ Check response headers for X-Powered-By, Server
  • ▸ Find API docs: /api/docs, /swagger, /graphql
  • ▸ Check JS files for API endpoints, tokens, secrets
  • ▸ nuclei -u target.com -t exposures/ for quick wins
5
MANUAL TESTING IN BURP SUITE

This is where the real money is. Automated scanners miss the majority of bugs. Set up Burp as your proxy and manually walk through every feature as a real user would.

  • ▸ Intercept every request — look at params, headers, cookies
  • ▸ Create 2 accounts — test IDOR by swapping IDs between them
  • ▸ Test every input field for XSS, SQLi, SSTI
  • ▸ Check every state-changing request for CSRF token
  • ▸ Try privilege escalation: user → admin endpoints
  • ▸ Burp Repeater + Intruder are your best friends
6
CONFIRM & DOCUMENT THE BUG

Before reporting — confirm it's real, reproducible, and actually impactful. Duplicate reports = $0. Document everything: steps, request/response, screenshots, impact.

Screenshot everything Save raw HTTP request/response Note the exact vulnerable parameter
7
WRITE A KILLER REPORT

Report quality directly affects payout. Clear, detailed reports with PoC get higher severity ratings and faster payment. Use the Report Builder tab → copy the template exactly.

// VULNERABILITY CLASSES — WHAT PAYS

These are the bugs that actually get paid. Learn each one deeply. PortSwigger Web Academy has free labs for all of these.

SQL INJECTION CRITICAL

Attacker injects SQL into input fields to dump databases, bypass auth, or RCE.

Where to look: login forms, search boxes, URL params, API endpoints with numeric IDs.

Quick test: Add ' to any input — SQL error = potential vuln.

Tool: sqlmap -u "URL?id=1" --dbs

→ Learn on PortSwigger

XSS HIGH

Inject JavaScript into pages viewed by other users. Steal cookies, sessions, deface pages.

Where to look: Any input that reflects back in the page — search, comments, profile fields, error messages.

Quick test: <script>alert(1)</script> in every field.

Stored XSS (persists for other users) pays more than Reflected.

→ Learn on PortSwigger

IDOR HIGH — MOST COMMON

Insecure Direct Object Reference — accessing other users' data by changing an ID.

Where to look: /api/users/1337 → try /api/users/1338. Order IDs, invoice IDs, file IDs.

How to test: Make 2 accounts. Access Account B's resources using Account A's session.

One of the most common paid bugs. Easy to find, easy to report.

→ Learn on PortSwigger

CSRF MEDIUM

Trick a logged-in user's browser into making unauthorized requests on their behalf.

Where to look: State-changing POST requests (password change, email change, money transfer) without CSRF tokens.

Quick test: In Burp, check if removing CSRF token or using a fake one still works.

→ Learn on PortSwigger

SSRF CRITICAL

Server-Side Request Forgery — make the server fetch internal URLs. Can expose AWS metadata, internal services, cloud credentials.

Where to look: Any input accepting a URL — webhooks, PDF generators, image importers.

Quick test: Try http://169.254.169.254/ (AWS metadata) as URL input.

→ Learn on PortSwigger

OPEN REDIRECT LOW–MED

Redirect users from trusted domain to attacker's site. Used for phishing, OAuth token theft.

Where to look: ?redirect=, ?next=, ?url=, ?return= params.

Quick test: Set value to https://evil.com — does it redirect?

Low payout alone but can chain with OAuth for High.

SUBDOMAIN TAKEOVER HIGH

Dangling DNS record pointing to a deprovisioned cloud service you can claim.

Where to look: Subdomains returning NXDOMAIN or "NoSuchBucket" / "There is no app here".

Tool: subjack, nuclei -t takeovers/

Can serve malicious content on a trusted domain. Easy money.

AUTH BUGS CRITICAL

Broken auth: weak JWT signing, password reset flaws, 2FA bypasses, session fixation.

Where to look: Password reset flow, JWT tokens (jwt.io to decode), OAuth callbacks.

Quick test: Check if JWT uses alg: none. Try using another user's reset token.

→ Learn on PortSwigger

// THE FASTEST PATH TO FIRST PAYOUT

BEGINNER TARGETS

Start with programs that have large attack surfaces, lower competition, and pay for low/medium severity.

  • HackerOne public programs — sort by "most reports"
  • Programs with wide scope (*.company.com)
  • Bug Bounty programs on Bugcrowd with "VRT" ratings
  • Avoid invite-only private programs until you have reps
QUICK WIN STRATEGY

These bugs are easy to find and still pay. Build your rep with these while learning harder stuff.

  • Subdomain takeovers — run nuclei takeover templates
  • IDOR on API endpoints — make 2 accounts, swap IDs
  • Exposed admin panels / .git directories
  • Missing security headers on sensitive pages
  • Open redirects — check every ?url= param
